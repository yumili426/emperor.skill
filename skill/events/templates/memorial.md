# 奏折模板

## 场景

皇帝御批奏折，处理日常政务。

## 流程

```
[Step 1] 呈递奏折
  太监："启禀万岁，今日共有 N 份奏折待批。"

[Step 2] 展示列表
  列出所有待批奏折（标题 + 提交者 + 紧急程度）

[Step 3] 皇帝选择
  选择要批阅的奏折（输入序号）

[Step 4] 批阅
  阅读奏折正文，做出决策

[Step 5] 结算
  记录决策，更新状态
```

## 奏折来源

### 1. 现实事件映射

将用户的真实生活事件转化为奏折：

| 现实场景 | 奏折标题 | 提交者 | 示例内容 |
|---------|---------|--------|---------|
| 工作任务 | 《请旨拨款修缮北境城墙》 | 工部尚书 | "臣请拨银三万两，修缮北境城墙，以防胡虏犯边。" |

| 朋友求助 | 《同乡知县求援疏》 | 户部尚书 | "某地知县上疏，称管辖之地遭旱灾，请求拨粮赈灾。" |
| 被批评 | 《御史弹劾疏》 | 都察院 | "臣弹劾某官玩忽职守，请旨查办。" |
| 加班赶工 | 《军情急报》 | 兵部尚书 | "北境急报，胡虏集结重兵，恐有大规模犯边。" |
| 项目上线 | 《工部竣工疏》 | 工部尚书 | "臣奏报，京师城墙修缮工程已竣工，请旨验收。" |

### 2. 系统自动生成

根据国家状况自动生成：

| 国家状况 | 奏折类型 | 频率 |
|---------|---------|------|
| 经济 < 50 | 户部请旨加税 | 每月 |
| 军事 < 50 | 兵部请旨增兵 | 每月 |
| 稳定 < 50 | 刑部请旨严刑 | 每月 |
| 民心 < 50 | 礼部请旨赈灾 | 每月 |
| 国库 > 20000 | 户部请旨拨款 | 每季 |
| 邻国关系恶化 | 兵部请旨备战 | 随机 |

## 批阅选项

| 选项 | 说明 | 影响 |
|------|------|------|
| A. 准奏 | 同意该请求 | 提交者忠诚度+5，执行效果 |
| B. 驳回 | 拒绝该请求 | 提交者忠诚度-5 |
| C. 交由内阁审议 | 让首辅团队讨论 | 效率取决于首辅能力 |
| D. 亲自处理 | 皇帝亲自决定 | 耗费精力，但控制力强 |
| E. 留中不发 | 暂不处理 | 无变化，但积压过多影响稳定 |

## 批阅后果

### 准奏后果

- 执行该请求的实际效果
- 提交者忠诚度+5
- 相关国家数值变化

### 驳回后果

- 提交者忠诚度-5
- 可能触发角色不满事件
- 多次驳回同一角色：忠诚度额外-10

### 交由内阁后果

- 效率 = 首辅能力 × 0.8
- 首辅忠诚度高：自动做出合理决策
- 首辅忠诚度低：可能做出损害皇帝利益的决策

### 亲自处理后果

- 控制力强，效果最优
- 但每次亲自处理消耗精力
- 精力不足时：稳定度-2

### 留中不发后果

- 暂时无影响
- 积压3份以上：稳定度-5
- 积压5份以上：稳定度-10，可能触发弹劾事件

## 现实事件映射规则

当用户提到现实事件时，自动转化为奏折：

```
用户说："今天加班到很晚"
  ↓
系统生成奏折：
  标题：《军情急报》
  提交者：兵部尚书
  正文："前线急报，胡虏集结重兵，我军将士日夜守城，疲惫不堪。请旨增派援军。"

用户说："老板批评我了"
  ↓
系统生成奏折：
  标题：《御史弹劾疏》
  提交者：都察院
  正文："臣弹劾某官玩忽职守，请旨查办。"
```

## 与早朝系统的区别

| 维度 | 早朝 | 奏折 |
|------|------|------|
| 触发方式 | 每日固定 | 随时可处理 |
| 互动形式 | 群臣奏报，集体决策 | 单独批阅，逐个决策 |
| 事件类型 | 重大事件、紧急事件 | 日常政务、次要事件 |
| 决策影响 | 较大，影响全局 | 较小，影响局部 |
| 处理时间 | 一次处理多个 | 逐个处理 |
